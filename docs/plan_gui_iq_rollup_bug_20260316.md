# Plan: Fix GUI IQ Rollup Bug

## Problem 
1. **IQ Rollup Bug**: In the R/Shiny GUI proteomics workflow, the module `mod_prot_qc_protein_rollup.R` performs peptide-to-protein rollup using the `iq` package. It writes the `peptide_s4` long data to a TSV, runs `iq::process_long_format()`, and then reads the result back with `vroom::vroom()`. It then tries to create a `ProteinQuantitativeData` S4 object. The S4 object's validity check (`func_prot_s4_objects.R`) throws the error: `"Samples in protein data and design matrix must be the same"` when processing the neurolincs parquet dataset. This happens for two possible reasons:
    - **Samples dropped by `iq`**: `iq` filters out peptides not meeting Q-value thresholds (`filter_double_less = c("Q.Value" = "0.01", ...)`). If a sample has no remaining valid proteins after IQ processing/filtering, it is dropped from the wide-format output. However, the `mod_prot_qc_protein_rollup.R` script passes the *original* `peptide_s4@design_matrix` directly to `ProteinQuantitativeData()`.
    - **Column Name Mangling**: `vroom::vroom()` defaults to `.name_repair = "unique"`. Additionally, `iq` or `readr` processes might alter sample names if they contain special characters, spaces, or start with numbers, transforming them into "syntactically valid" R names. This causes a mismatch with the exact strings in the design matrix `Run` column.
2. **Composite QC Figure Saving**: The `_composite_QC_figure` generated during the normalization step (`mod_prot_norm.R`, `mod_lipid_norm.R`, `mod_metab_norm.R`) is currently only saved as a PNG. The requirement is to save this figure as a PDF, a PNG, and an RDS file containing the R plot object, using the `savePlot()` function.

## Proposed Changes

### 1. Robust Column Name Preservation (Internal Aliasing)
To proactively prevent column name mangling by `iq` or `vroom` without forcing the user to change their original sample names, we will implement an internal safe-mapping strategy in `R/mod_prot_qc_protein_rollup.R`:
- **Map Generation**: Create safe, standardized aliases (e.g., `s_001`, `s_002`) for every unique sample name in the design matrix.
- **Alias Application**: Replace the actual sample names in the long-format `peptide_data_for_iq` dataframe with their aliases *before* saving to the temporary TSV for IQ.
- **IQ Execution**: Run `iq::process_long_format()`. The tool processes valid, safe names, avoiding any internal parsing issues.
- **Alias Restoration**: Read the resulting file back with `vroom(..., .name_repair = "minimal")`. Use the mapping dictionary to safely translate the column names (`s_001`, `s_002`) back to the user's authentic, original sample names.

### 2. Dynamic Design Matrix Filtering for Dropped Samples
If specific samples drop out completely due to strict Q-value filters in `R/mod_prot_qc_protein_rollup.R`:
- Interrogate the restored column names of the `protein_log2_quant` after restoring aliases to identify which exact samples survived.
- Compare these against the original `peptide_s4@design_matrix`.
- If a mismatch is detected, filter the design matrix passed to the `ProteinQuantitativeData` constructor so it only contains the surviving samples.
- Trigger a `shiny::showNotification` showing a warning to the user explicitly detailing which samples were dropped during rollup.

### 3. User Control for Sample Name Management
To provide users with more control over how names are handled across the whole application:
- Recommend implementing a "Sanitize Sample Names" feature in the upstream **Data Import** module. 
- Using `janitor::make_clean_names()`, this option would allow users to proactively clean their sample IDs at the source. This ensures best practices are followed and mitigates issues not just in `iq` rollup, but across downstream linear modeling (e.g., `limma` contrasts) and plotting frameworks that struggle with spaces or special characters.

### 4. Updating Composite QC Figure Saving
Modify the `generateCompositeFromFiles` function implementation in `R/mod_prot_norm.R`, `R/mod_lipid_norm.R`, and `R/mod_metab_norm.R` (and check `R/func_general_filemgmt.R` if applicable) to leverage the `savePlot` function for creating the final file, ensuring it produces the `.pdf`, `.png`, and `.rds` versions:
- Currently, `generateCompositeFromFiles` combines PNG files from disk and saves the result directly.
- The function uses `magick` and `gridExtra` (via `grid::grid.draw()`) to piece together the images. 
- *Implementation Note*: Since the composite figure is generated by appending images and creating a standard plot output, we must ensure it can be passed to `savePlot` which requires a `ggplot` (or compatible `gg` object) to save to RDS. If `generateCompositeFromFiles` produces a plot that `ggsave` can handle, we will replace the direct file saving with `savePlot()`.
- Updates will target:
  - `R/mod_prot_norm.R`: `composite_path <- file.path(qc_dir, "composite_QC_figure")` (without extension to let `savePlot` handle it). Update the `generateCompositeFromFiles` call to return the grob/plot, then use `savePlot(composite_plot, qc_dir, paste0(omic_type, "_composite_QC_figure"))`.
  - Repeat the above for `R/mod_lipid_norm.R` and `R/mod_metab_norm.R`.
